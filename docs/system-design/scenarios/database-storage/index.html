Database & Storage Playbook

Structured design decisions around logical normalization, sharding keys, indexing, CDC pipelines, and time-series architectures.

Scenario 1: Architectural Decision for Database Normalization in a Scalable System​

Dilemma Statement:

Architectural Decision for Database Normalization in a Scalable System Q: As a software engineer at a growing e-commerce platform, Sarah is redesigning the database schema to improve performance and maintainability. She needs to decide on a database normalization level that balances query efficiency and data integrity. Which architectural approach should Sarah consider?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Partially denormalizing the database for read-heavy operations to reduce joins and improve query performance.

Detailed Analysis & Trade-offs:​

Explanation In this scenario, partially denormalizing the database for read-heavy operations is the best option. It helps reduce the number of joins, improving query performance without sacrificing data integrity. Full normalization to BCNF might lead to complex joins and performance issues, while opting for eventual consistency can introduce data inconsistency problems. Sharding focuses on horizontal scalability but is not directly related to database normalization

Scenario 3: Maya is a backend architect for a global e-commerc...​

Dilemma Statement:

Maya is a backend architect for a global e-commerce catalog service. Traffic is strongly read-dominated: 150k reads/sec (searches, product pages, personalization) and only ~100 writes/sec (price updates, inventory adjustments, product edits). Reads include complex denormalized queries (joins, aggregations) and must serve <100ms 99th-percentile globally. Some writes (inventory, price) must be authoritative and never allow oversell; other writes (product description edits, tags) can be slightly delayed in the UI. The team must pick an architecture that scales reads horizontally across regions, keeps write latency reasonable, keeps the catalog highly available, and provides clear operational visibility into staleness. Which architectural approach is the best primary strategy for this workload?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option C (CQRS with event-driven read models) is the best primary strategy for a read-heavy, globally served catalog with a small number of authoritative writes. Reasons: - Read/write separation: CQRS lets the write path remain transactional and authoritative for critical updates (inventory, price) while read models are independently optimized (denormalized, indexed, precomputed) for the heavy, complex read patterns. That lets read stores serve queries under the latency target without making writes pay for complex read indexes. - Horizontal scale and regional placement: materialized read views can be replicated or rebuilt into regionally located stores (Elasticsearch, Cassandra, read-optimized Postgres replicas) and cached locally to meet 99th-percentile latency and availability goals. - Operational observability and safety: an event log gives a clear source of truth and makes it straightforward to measure and alert on read model lag, retry/repair failed projections, and replay events if needed. Critical writes can be handled synchronously on the write side (and optionally synchronously update a small critical cache), while other read-side updates remain eventual. - Failure and fault tolerance: read-models are rebuildable from events; you can design idempotent handlers and checkpoints. Downsides are added complexity (event schema, projections, operations) and eventual consistency for reads, so you must surface staleness information to clients and design compensations for strict invariants (like using a short synchronous reservation path for inventory during checkout). Why the others are less suitable: - A (async DB replicas): cheaper initially but leaves you with the original schema and indexes tuned for writes or OLTP, not complex denormalized reads. Replication lag leads to unpredictable staleness; replicas don’t let you change storage shape to accelerate complex queries. - B (global write-through cache): forces every write to synchronously update cache and DB, increasing write latency and coupling writes to cache availability. A single global cache cluster is a potential bottleneck and failure domain across regions. Also maintaining strong consistency on critical writes with a distributed cache is hard and expensive. - D (sharding): sharding can scale both reads and writes, but it doesn’t solve the problem of complex denormalized queries (cross-shard joins, aggregations) and increases operational complexity dramatically (rebalancing, cross-shard consistency). For mostly-read workloads, it’s usually heavier-handed than creating read-optimized models. Implementation notes for C: keep a small synchronous write path (single-writer or transactional microservice) for inventory/price; emit events to a durable log (Kafka), build materialized views per region, use local caches for hot items, track and alert on projection lag, and design idempotent projection handlers and compensating flows for critical invariants. This approach gives the best balance of read scalability, availability, and predictable latency while keeping authoritative writes correct.

Scenario 4: Maya’s query bottleneck: subqueries or precompute?​

Dilemma Statement:

Maya is a backend engineer at a payments platform. A real-time dashboard endpoint needs to return per-customer metrics (last 30-day transaction counts, top 3 merchants, rolling spend percentiles) under 200 ms. Today the endpoint runs a single complex SQL that nests several subqueries and multiple CTEs over large tables. Under load the DB CPU spikes, p99 latency jumps, and dashboards time out. Maya must choose a path forward that balances latency, scalability, consistency, and operational complexity. Which architectural choice is the best fit for production?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option C is the best fit for Maya’s production needs. The dashboard is read-heavy and requires low p99 latency; repeatedly running large nested subqueries and CTEs over big tables is expensive and hard to scale. Precomputing denormalized summaries or using materialized views (with incremental refresh where possible) moves expensive aggregation off the critical path, lets you index the summary for fast reads, and drastically reduces CPU and latency for the dashboard. Use CDC or background workers to keep summaries reasonably fresh, instrument staleness metrics, and provide fallback paths for very recent data. This approach improves scalability and availability at the cost of eventual consistency and extra operational work (maintaining update pipelines, handling retries and schema changes), which is an acceptable trade for dashboards and read SLAs.

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

A (scale DB vertically / read replicas): Vertical scaling and replicas can help short-term but are expensive and limited. Read replicas don’t fix heavy aggregation cost if each request still does full scans or expensive joins. You also increase blast radius and operational cost; this approach delays but doesn’t solve the root cause. B (many per-request CTEs/temp tables + hints): Breaking queries into CTEs or temp tables can sometimes help clarity, but CTEs in some databases act as optimization fences (e.g., older Postgres behavior) and can prevent the planner from optimizing across steps, making things worse. Per-request partial materialization still costs CPU and I/O each time. Hints and tunings are brittle and DB-version dependent. D (move computation to app/service layer): Offloading to app memory or an analytics service can reduce DB load, but you trade that for increased latency (networking + compute), higher memory/CPU on services, and more complex horizontal scaling and fault tolerance. For per-request, synchronous aggregation across large volumes is still slow. An async precompute pattern combined with small, optimized reads is typically more efficient. Implementation notes for C: choose incremental materialized views or summary tables with change data capture (Debezium, WAL tailing) or event-driven jobs to apply updates; index the summary columns used in queries; expose a clear SLA for staleness (e.g., <30s); instrument metrics for refresh lag and fallback rates; include idempotent workers and retry logic for fault tolerance.

Scenario 5: Maya's telemetry bottleneck: scale raw metrics without breaking dashboards​

Dilemma Statement:

Maya is building a backend for telemetry from millions of IoT devices. Each device sends metric points every 10 seconds. Requirements: sustain very high ingest rate (tens of thousands of writes/sec), serve low-latency recent-time dashboards (sub-second to a few seconds), support ad-hoc historical analytics (days to years), keep 1 year of data but keep full fidelity for 7 days and downsample older data, and tolerate regional failures. Maya needs to pick an overall time-series data modeling and ingestion architecture that balances scalability, latency, availability, and operational cost. Which architectural choice best fits these needs?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

B is the best fit because it decouples high-rate ingestion from query serving and long-term analytics, which addresses the core trade-offs: scalability, low-latency recent reads, and cost-effective historical storage. Writing first to a durable log (Kafka) makes ingestion highly available and allows producers to be fire-and-forget; it also buffers spikes and enables replay for reprocessing. Stream processing provides low-latency aggregates and enrichment without blocking writes. Storing recent, query-heavy data in a purpose-built TSDB (or an analytical store tuned for point and time-range queries) gives sub-second dashboard performance. Persisting raw events to object storage in partitioned Parquet files provides cheap, durable, queryable cold storage for large-scale analytics and for rebuilding derived data if needed. Downsampling/rollups can be done asynchronously (stream jobs or batch jobs), keeping write latency low and controlling storage costs. This architecture also supports regional failover patterns and independent scaling of components (broker, stream processors, TSDB, object storage).

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

A (single Postgres with synchronous writes): Simple and consistent but won't scale easily to tens of thousands of writes/sec without very complex sharding and heavy ops. Synchronous writes and triggers for aggregates add write latency and increase contention. Vertical scaling and master failover are limited for global availability and cost-efficient long-term storage. C (single TSDB cluster with synchronous downsampling): A specialized TSDB can handle time-series workloads, but doing downsampling synchronously at ingest increases write latency and backpressure under load. A monolithic TSDB as the sole store becomes a single scaling and operational bottleneck (and a risky blast radius). Some TSDBs also have less mature long-term cold storage integration compared with the log+object-store pattern. D (Cassandra with time-bucketed partition keys): Cassandra shines for high write throughput and availability, but range/time-based analytics and ad-hoc historical queries are harder and typically inefficient. Materialized views and application-side aggregations introduce complexity and eventual consistency; repairing and compaction behavior across time-partitioned data can complicate downsampling and retention. Operationally, implementing full-fidelity long-term storage and flexible analytics is more work compared to an object-store-based cold tier. Trade-offs and notes: Option B does add operational complexity (running Kafka, stream processors, and multiple data stores). But it provides cleaner separation of concerns, predictable scaling characteristics, natural support for tiered storage and retention policies, and the ability to replay/rebuild derived data. For dashboards that require strict strong consistency per single write, you must accept small windows of eventual visibility between systems or implement read-path aggregation from the TSDB that’s fed by the stream jobs. Overall, for large-scale telemetry with different SLAs for recent vs historical data, the log + streaming + hot TSDB + cold object store pattern is the most practical and robust choice.

Scenario 6: Maya's Metrics Dilemma: Choosing a Timeseries Storage Strategy​

Dilemma Statement:

Maya is a backend architect building a multi-tenant telemetry platform that ingests 100k samples/sec, serves 500 tenants, and must retain raw data for 90 days and aggregated data for 1 year. Queries are a mix of recent real-time dashboards (sub-second to a few seconds) and large-range analytics (minutes to hours). SLOs: 99.95% read availability for dashboards, 99.9% write availability, and query latencies under 5s for common aggregates. Costs must be controlled. Maya needs to pick a time-series data modeling and storage architecture that balances ingestion throughput, query latency, availability, consistency, and storage cost. Which approach should she choose?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option B is the best fit for Maya's constraints because it matches the common, proven pattern for large-scale time-series systems: use an append-only ingestion path with durable buffering, a fast hot store for recent queries, and a cheaper cold store for long-range analytics. Key reasons: - Ingestion and availability: buffering writes in Kafka (or similar) decouples producers from backend storage and enables high sustained ingest (100k+/sec) while keeping write availability high and avoiding backpressure from downstream compactions. - Scalability: sharding by tenant and time lets you spread write/read load evenly, reassign hot shards, and scale individual components independently (brokers, hot nodes, OLAP nodes). - Latency: a write-optimized hot store (Cortex/Thanos/Influx/ClickHouse in a write-friendly mode) keeps recent partitions on fast storage so dashboards hit sub-second to few-second latencies. Cold data moved into a columnar store or object-store-backed engine gives cost-effective long-term retention and fast aggregates over large ranges. - Cost and operability: async downsampling and compaction reduce storage growth and improve query performance for long ranges. Eventual consistency is acceptable for metrics in most cases and buys much higher availability and throughput. Why the others are less suitable: - Option A (single Postgres with partitions): technically possible for modest scale, but relational DBs struggle with sustained high ingest and cardinality of tags. Synchronous transactional writes and heavy indexing will increase write latency and storage overhead. Read replicas help but don't solve hot partitions or high cardinality tag queries, and operational scaling becomes painful. - Option C (CP-style synchronous replication): gives strong consistency but at the cost of higher write latency and lower availability during partitions. For high ingest time-series data you usually prefer availability and throughput over strict cross-shard consistency; synchronous replication will become a bottleneck. - Option D (per-tenant clusters): provides isolation but is operationally expensive and wasteful when you have hundreds of tenants. Small tenants will underutilize resources, upgrades and schema changes become harder, and cross-tenant analytics get complex and slow. Practical notes when implementing B: - Design a narrow schema (time, metric id, value, tags) and push tag cardinality into an indexed tag store or inverted index rather than wide relational columns. - Keep recent partitions hot (in-memory or fast SSD), compact older partitions and store them in object storage with metadata for fast reads. - Implement throttling, backpressure, and multi-tenant resource limits to avoid noisy-neighbor effects. - Provide graceful eventual-consistency guarantees in the API (timestamps and ingestion windows) and expose observability for lag, purge, and compaction jobs.

Scenario 7: Kai must pick a replication model for a global wallet — accuracy vs latency​

Dilemma Statement:

Kai is a backend engineer building a global user wallet service used for instant payments and balance lookups. The service must: (1) never allow double-spend or incorrect balances, (2) serve low-latency reads from users in multiple regions, and (3) remain available during single-region failures when possible. Kai must choose a replication and consistency architecture for the ledger shards. Which approach best balances strict correctness with practical global latency and operational complexity?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option B is the best practical choice for a global wallet that must avoid double-spend while keeping latency and operational complexity reasonable. Sharding the ledger by account (or wallet) and using a small consensus group per shard gives strong, linearizable writes for that shard because updates require a majority commit. Placing the leader near the shard's primary traffic minimizes typical write latency. Reads that must be strongly consistent are served from the leader; when stale reads are acceptable (analytics, balance previews), followers or read replicas provide low-latency local access. This model is production-proven (many distributed databases and services use per-shard Raft groups) and allows operational controls: rebalancing shards, moving leaders for hotspots, monitoring replica lag, and handling leader election during failures without risking double-spend.

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

  • Option A (global synchronous write to every region) gives strong global consistency but at the cost of high write latency (every write waits for the slowest region) and reduced availability during inter-region network problems. It also scales poorly operationally because every write involves many replicas.
  • Option C (multi-master with CRDTs/eventual) provides low-latency local writes but sacrifices strong correctness semantics. CRDTs are great for commutative counters or mergeable data, but money transfers and wallet balances require strict, non-commutative ordering and no lost updates. Designing correct conflict resolution for financial invariants is extremely hard and risky.
  • Option D (single primary with async replication) centralizes correctness on one node, but async replicas can lag and you still need to route every write to the primary, creating higher latency for remote users and a single-region availability dependency. Attempting to overlay distributed transactions or compensating logic to hide lag increases complexity and still doesn't guarantee no double-spend under partitions.

Operational and observability notes for option B: track per-shard commit latency, leader election frequency, follower replication lag, and QoS for leader locality. Build tooling for shard rebalancing and automated leader transfer to reduce hot-spot risk. For global strong consistency requirements that must span many shards in a single transaction, consider higher-level design changes (sagas, two-phase commit across shard leaders) and reduce cross-shard transactions where possible.

Scenario 8: Diego's dilemma: How to enforce data types at 50k writes/sec?​

Dilemma Statement:

Diego is the lead backend engineer for an industrial IoT platform that collects sensor readings from 100k devices at peak, producing ~50k writes/sec. Each reading must include a device_id, timestamp, metric name, unit (e.g., C or F), and numeric value. Business requirements: (1) low write latency and high availability, (2) enforce basic type and range constraints (e.g., numeric value within sensor-specific limits, valid units), (3) support schema evolution as new device types arrive, (4) allow fast aggregations for dashboards and alerts, and (5) keep strong auditability of raw incoming data. Diego must pick an architecture for ingestion and storage that balances throughput, consistency, observability, and constraint enforcement. Which approach best meets these goals?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option D is the best practical balance for Diego's requirements. Using a durable message bus plus schema registry enforces typed messages and safe schema evolution at ingestion time without making the primary storage a single bottleneck. Stream processors let you run deterministic validation (units, ranges, device-specific checks) and normalization at scale, then write validated records to a storage optimized for the query patterns (time-series DB for aggregations, partitioning, indexes). The raw topic persisted to a data lake preserves auditability and supports reprocessing if validation rules change. This architecture maximizes write throughput and availability (Kafka is write-optimized and tolerant to consumer/backpressure), provides clear ownership of schema (schema registry), keeps observability (consumer lag, schema versions, processing metrics), and isolates constraint enforcement outside the transactional DB so you can scale reads and writes independently.

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

  • Option A (single Postgres with strict DB constraints) gives strong integrity but becomes a scalability and availability bottleneck at 50k writes/sec. Vertical scaling and partitioning help, but DB-level validation creates latency and can increase contention. Schema evolution across many device types is harder and more disruptive when every change requires DB migrations.
  • Option B (schemaless NoSQL with deferred validation) scales writes and is highly available, but pushing validation to reads or slow background jobs sacrifices data quality and increases query complexity and latency. You lose the immediate guarantees needed for alerting and downstream consumers, and fixing bad data later is costly.
  • Option C (distributed SQL with client-side validation) gives strong consistency and horizontal scaling, but client-side validation is brittle (clients may be inconsistent or buggy). Distributed SQL also tends to have higher write latencies and cost than an append-only log plus specialized stores. While this option is viable for some workloads, it mixes the complexity of ensuring every producer is correct with the cost of strong transactional semantics at high ingest rates.

Trade-offs and practical notes: choose D when you need high ingest throughput, schema evolution, auditability, and low-latency analytical queries. Implement a schema registry and enforce producer-side and ingest validation (reject or route invalid messages to a dead-letter topic). Use compacted topics to store device metadata and enrich in stream processing. Persist raw and validated streams separately so you can reprocess when rules change. Monitor schema versions, validation error rates, consumer lag, and end-to-end latency to ensure constraints are actually enforced in production.

Scenario 9: Maya needs sub50ms reads; indexonly scan or something else?​

Dilemma Statement:

Maya is a backend engineer working on a globally distributed user-profiles service. The profiles table is wide (hundreds of columns, large JSON blobs), but 80% of reads ask for only 3 small fields (user_id, display_name, avatar_url) filtered by email or username. Traffic is high (peak 8k qps reads, 1k qps writes). The DB is a sharded SQL cluster (MVCC semantics, autovacuum enabled), and cross-shard fetches add extra latency. Current queries do full row lookups and often fetch heap pages, leading to higher tail latency and extra network/io. You need to choose an architecture to get stable sub-50ms read latency at peak while keeping the system reliable and maintainable. Which approach is best?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option A is the best fit for this scenario because the hot queries request only a few small columns from a very wide table. Creating covering indexes (or indexes with included columns) lets the database return rows using the index pages alone (index-only scans), avoiding expensive heap fetches and cross-shard row retrieval. In a sharded SQL cluster with MVCC, index-only scans are also more network-friendly since you read smaller index pages per shard. To make this practical you should: include exactly the needed columns (or use partial indexes to reduce size), tune autovacuum/visibility-map settings so the planner can rely on index visibility, and instrument write latency and index bloat because extra indexes increase write amplification and storage. This preserves strong consistency and keeps reads low-latency without adding a separate system or eventual-consistency surface.

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

  • B (async denormalized read table): This reduces read latency but introduces eventual consistency and more operational complexity (background jobs, CDC, reconcilers). For use-cases that need near-immediate consistency on profile reads, this adds risk and debugging surface. It's a good fallback if indexes create unacceptable write load.
  • C (separate columnar/OLAP store): Column stores excel at analytics and large scans, but they add ETL latency and operational overhead and usually can't meet strict sub-50ms, strongly consistent OLTP read requirements. Not appropriate for high-QPS, low-latency point-lookup workloads.
  • D (global in-memory cache): Caching can reduce load but brings cache invalidation complexity on writes, cold-start latency, and possible staleness. For data that must be strongly consistent or updated frequently at 1k qps, keeping correctness and coherent invalidation across a global fleet is hard; caching is best as a complementary optimization after indexing.

In short, for high-QPS point lookups returning a few columns from a wide MVCC-backed table, covering indexes with index-only scans give the best trade-off of latency, consistency, and operational simplicity. Monitor index size, write throughput, autovacuum behavior, and be prepared to use partial indexes or selective denormalization if write cost becomes unacceptable.

Scenario 10: Aiden's Denormalization Dilemma: fast reads or strict correctness?​

Dilemma Statement:

Aiden is a backend engineer responsible for the user profile and feed service at a fast-growing social app. The read pattern is heavy (10k RPS) and must target <50 ms tail latency with 99.99% read availability. Writes are much lower (200/s) but sometimes include critical fields (account balance, subscription status) that must be strongly correct for billing and access control. Currently the data is normalized in a single relational DB and reads do multiple joins across user, preferences, and aggregate tables, causing high read latency and DB load. Aiden needs to choose an approach to reduce read latency and scale reads while keeping the system maintainable and safe for the critical fields. Which architectural approach best balances low-latency/high-availability reads and the need for strong correctness on critical fields?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

**D is the best practical choice because it matches the differing requirements: low-latency, highly available reads for UI data and strong correctness for sensitive fields. Practical pattern: keep a transactional path for critical fields (single-source-of-truth DB, or a small service that uses transactions or consensus for updates), while using an outbox or CDC to publish changes for non-critical data into a stream (Kafka). Stream processors or consumers update a denormalized read store (key-value DB, search index, or cache) to serve low-latency reads. Implement idempotent consumers, versioned writes, dead-letter handling, and periodic reconciliation jobs to repair drift. Use monitoring and per-field SLAs to detect stale data and route critical reads to the authoritative path when necessary.

Why not A: Scaling the primary relational DB and doing joins at read time keeps strong consistency but fails the latency and cost targets at very high read rates. Read replicas help but can't always meet <50ms tail latency under join-heavy workloads and increase operational complexity (replication lag, hotspotting).

Why not B: Synchronous denormalization (updating read copies inside the write transaction) yields strong consistency but pushes the cost to writes; higher write latency, larger transactions, and more potential contention. It complicates sharding and cross-partition transactions and worsens availability during write-side failures. For mostly-read workloads, this over-weights correctness for data that doesn’t need it.

Why not C: Full async denormalization gives excellent read latency and availability, but it exposes critical fields to unbounded staleness and makes correctness guarantees hard. For billing/access data, eventual consistency is often unacceptable. Also, an async-only architecture can be harder to reason about without mechanisms for reconciliation and observability.

Trade-offs and operational notes: Implement transactional outbox to avoid lost events on failure. Use compacted topics, idempotent writes to the read store, and include monotonically increasing version numbers or vector clocks to prevent regressing state. For critical reads, query the authoritative store or perform a quick consistency check (e.g., read-through cache with validation). Add end-to-end tracing and SLAs per field so you can detect and mitigate stale reads. This hybrid approach minimizes read latency and cost while containing complexity where strict correctness truly matters.**

Scenario 11: Ava battles a sudden hotkey that’s melting the cache; what architecture wins?​

Dilemma Statement:

Ava is a backend engineer responsible for a user-profile service used by the product feed. The service is read-heavy (95% reads), reads are latency-sensitive (p95 < 50 ms), and slight staleness up to a few seconds is acceptable. Recently a specific profile ID became extremely popular (a hot key) and caused repeated cache misses and DB overload during traffic spikes. The system currently uses a simple memcached cluster with consistent hashing and a primary SQL DB. Ava must choose an architectural fix that prevents single-key hotspots from degrading overall latency and availability, while keeping cost and operational complexity reasonable. Which approach should she pick?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

Option C is the best fit for this scenario. It targets the root cause (hot-key concentration) without creating a new global bottleneck. Key elements: 1) Hot-key detection; use per-key QPS/miss-rate metrics and an automated threshold to mark keys as hot. 2) Isolation and replication; move hot keys to a hot tier or replicate that key onto multiple cache nodes so reads are spread and one node’s saturation doesn’t impact others. 3) Request coalescing (single-flight); collapse concurrent cache misses into one DB fetch/refresh to prevent stampedes. 4) Async write-behind / controlled TTL; since a few seconds of staleness is acceptable, allow asynchronous updates or slightly longer TTLs so refreshes don’t block reads or writes. Together these reduce DB load, keep p95 latency low, and avoid single-point failures. Trade-offs: added complexity (hot-key detection, replication, routing), potential for short-term staleness, and operational work to autoscale the hot tier. These are acceptable given the constraints. Why the others are less suitable: A (centralized lock) serializes traffic for the hot key and creates a single point that can delay thousands of requests and violate p95 SLOs; it also makes the system less available if the lock service has issues. B (scale-up and write-through) relies on larger instances and synchronous DB writes; this increases cost, doesn’t fix stampedes on cache warm-up, and will increase write latency and DB load during spikes. D (very long TTLs + stale-while-revalidate) reduces DB load but risks serving stale data beyond the acceptable window for some flows, and it doesn’t prevent the initial stampede when the key first becomes hot; background refreshes can still collide and cause bursts. In short, C provides a targeted, production-proven pattern (hot-key isolation + single-flight + bounded staleness) that balances latency, availability, and consistency for read-heavy, slightly-stale-tolerant workloads.

Scenario 12: Maya's Timeseries Scaling Puzzle: Hot vs Cold, Realtime vs Batch​

Dilemma Statement:

Maya is the backend architect for a SaaS monitoring product. Incoming telemetry comes from ~100k agents, each sending 1–5 time-stamped metrics every 10s (roughly 10k–50k writes/sec). Retention is 2 years for raw data, but dashboards and alerts need sub-second reads for the most recent 1–6 hours and complex historical analytics on daily/weekly windows. She must pick an architecture for storing and serving time-series data that meets these requirements while controlling cost and operational complexity. Which architecture should she choose?

Click to expand recommended architecture & tradeoffs

Recommended Solution:​

**Ingest via a durable append log (Kafka) -> stream processing (Flink/Kafka Streams) to produce near-real-time materialized aggregates and downsampled streams. Persist raw samples cold in object storage (Parquet on S3) partitioned by metric/time for long-term analytics, and keep hot recent data and fast rollups in a time-series optimized OLAP store (ClickHouse/TimescaleDB) for low-latency dashboards and alerts.

TL;DR: Understanding the core trade-offs and architectural patterns helps you make informed system design decisions.

Option C is the best fit for Maya's constraints because it cleanly separates concerns: high-ingest durability and backpressure handling via Kafka, real-time computation and downsampling via stream processors, cheap and scalable long-term storage in columnar files on object storage, and a read-optimized time-series OLAP store for low-latency dashboards and alerts. This design scales horizontally for write and read load, limits hot storage size (reducing cost), supports complex historical queries by reading Parquet partitions, provides fault tolerance (Kafka + durable object storage + idempotent stream processing), and gives observable pipelines (consumer lag, processing metrics, S3 object listing). Trade-offs include added operational complexity (running Kafka + stream processors + multiple stores) and eventual consistency between raw and materialized views, which you can manage with monotonically increasing offsets, idempotent writes, and SLA'd freshness for dashboards.**

Detailed Analysis & Trade-offs:​

Why the others are less suitable:

  • A (single Postgres): While familiar and transactional, a single Postgres cluster will hit write throughput and storage limits at these rates. Vertical scaling is expensive and brittle. Even with partitioning and replicas, long-term retention of raw samples (2 years) and heavy analytical scans will make queries slow and backups painful. Postgres also struggles with very high-cardinality, time-partitioned write patterns at scale.
  • B (Cassandra/Scylla): This is strong for high write throughput and TTL-driven retention, and it provides good availability. But range/time-window queries and ad-hoc analytics are harder and often inefficient; implementing aggregations and joins is limited or requires heavy client-side work. Secondary index and wide time-range scans can be expensive. It also provides eventual consistency semantics that complicate alerts and recent-state reads unless you add more complexity.
  • D (RedisTimeSeries as primary): In-memory solutions give excellent latency but are costly for multi-year retention and risky as source-of-truth (memory limits, failover complexity, RTO/RPO concerns). Asynchronous archival can lead to data loss on failures between writes and archive, and batch overnight archiving increases latency for analytics. Redis is best as a cache or for short-term hot data, not the primary store for long retention at this scale.

Operational notes for C: partition Parquet by (metric_namespace, date/hour) to prune scans; keep SLA'd hot window (e.g., last 6 hours) in ClickHouse/TimescaleDB with frequent materialized rollups for dashboards; make stream jobs idempotent and track offsets for replay; use TTL/compaction policies on OLAP store to avoid storing duplicate rollups. Monitor consumer lag, commit latencies, and S3 latency for cold reads.
